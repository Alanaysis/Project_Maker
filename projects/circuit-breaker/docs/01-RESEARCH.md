# 研究文档: 熔断器模式

## 1. 背景与动机

### 1.1 分布式系统中的问题

在微服务架构中，服务之间的依赖关系复杂，一个服务的故障可能通过调用链传播，导致整个系统崩溃。这种现象被称为**级联故障**（Cascading Failure）。

### 1.2 熔断器模式的起源

熔断器模式（Circuit Breaker Pattern）由 Michael Nygard 在其著作《Release It!》中提出，灵感来源于电路中的断路器。当电流过大时，断路器会自动断开，保护电路不受损坏。

## 2. 核心概念

### 2.1 状态机

熔断器有三种状态：

1. **关闭状态（Closed）**
   - 正常处理请求
   - 统计失败次数和失败率
   - 当失败达到阈值时，切换到打开状态

2. **打开状态（Open）**
   - 直接拒绝所有请求
   - 返回降级响应或错误
   - 经过超时时间后，切换到半开状态

3. **半开状态（Half-Open）**
   - 允许部分请求通过
   - 测试服务是否恢复
   - 如果成功，切换到关闭状态
   - 如果失败，切换到打开状态

### 2.2 状态转换图

```
                   失败达到阈值
    关闭 (Closed) ──────────────→ 打开 (Open)
         ↑                            │
         │                            │ 超时
         │                            ↓
         └─────────────────────── 半开 (Half-Open)
                   成功达到阈值
```

## 3. 关键参数

### 3.1 失败阈值（Failure Threshold）

触发熔断的连续失败次数。例如，设置为5表示连续失败5次后触发熔断。

### 3.2 成功阈值（Success Threshold）

从半开状态恢复到关闭状态所需的连续成功次数。例如，设置为3表示连续成功3次后恢复。

### 3.3 超时时间（Timeout）

从打开状态切换到半开状态的等待时间。例如，设置为30秒表示30秒后尝试恢复。

### 3.4 失败率阈值（Failure Rate Threshold）

触发熔断的失败率。例如，设置为0.5表示失败率达到50%时触发熔断。

### 3.5 最小请求数（Minimum Requests）

计算失败率所需的最小请求数。例如，设置为10表示至少需要10个请求才会计算失败率。

## 4. 降级策略

### 4.1 默认降级

返回预设的错误信息或默认值。

### 4.2 缓存降级

返回缓存的数据，保证服务可用性。

### 4.3 静态降级

返回固定的静态数据。

### 4.4 组合降级

组合多种降级策略，按优先级执行。

## 5. 实现考虑

### 5.1 并发安全

熔断器需要支持并发访问，必须使用锁机制保护状态。

### 5.2 指标统计

需要统计请求数、成功数、失败数、失败率等指标。

### 5.3 时间窗口

某些实现使用滑动时间窗口统计指标，而不是简单的累计统计。

### 5.4 事件通知

可以添加事件监听器，在状态变化时触发回调。

## 6. 应用场景

### 6.1 远程服务调用

调用第三方API或微服务时，防止服务不可用导致的级联故障。

### 6.2 数据库访问

数据库连接池耗尽或数据库故障时，快速失败并降级。

### 6.3 缓存访问

缓存服务不可用时，降级到直接访问数据库。

### 6.4 消息队列

消息队列故障时，降级到同步处理或丢弃消息。

## 7. 相关模式

### 7.1 重试模式（Retry Pattern）

与熔断器配合使用，在熔断器关闭时进行重试。

**指数退避（Exponential Backoff）：**
- 第1次重试等待 baseInterval
- 第2次重试等待 baseInterval * multiplier
- 第3次重试等待 baseInterval * multiplier^2
- 以此类推，直到达到 maxInterval

**抖动（Jitter）：**
- 在退避间隔基础上添加随机偏移
- 避免多个客户端同时重试导致"惊群效应"
- 典型实现：`interval/2 + random(0, interval/2)`

### 7.2 舱壁模式（Bulkhead Pattern）

隔离不同服务的调用，防止故障传播。

### 7.3 超时模式（Timeout Pattern）

设置调用超时，避免长时间等待。

### 7.4 限流模式（Rate Limiting）

限制请求速率，保护服务不被过载。

**固定窗口限流（Fixed Window）：**
- 在固定时间窗口内限制请求数
- 实现简单，但窗口边界可能出现突发流量

**滑动窗口限流（Sliding Window）：**
- 使用时间戳记录每个请求
- 精确控制任意时间窗口内的请求数
- 内存开销较大

**令牌桶限流（Token Bucket）：**
- 以固定速率向桶中添加令牌
- 请求消耗令牌，桶空时拒绝
- 支持突发流量（桶容量 > 速率）
- 最常用的限流算法

## 8. API 网关中的应用

### 8.1 网关层防护

在 API 网关中，熔断器、限流器和重试机制通常组合使用：

```
请求 → 限流检查 → 熔断器检查 → 重试调用 → 响应
         ↓              ↓            ↓
       拒绝(429)    降级响应     最终失败
```

### 8.2 微服务调用链

在微服务调用链中，每个服务调用都应配置独立的熔断器：

```
API Gateway
  ├── User Service (熔断器 + 限流)
  ├── Order Service (熔断器 + 限流)
  └── Payment Service (熔断器 + 限流)
```
