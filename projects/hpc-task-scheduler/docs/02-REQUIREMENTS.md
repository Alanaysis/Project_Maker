# 需求分析

## 1. 项目概述

### 1.1 项目名称
HPC Task Scheduler（高性能计算任务调度系统）

### 1.2 项目目标
实现一个支持任务提交、资源分配、作业隔离的任务调度系统，用于学习 HPC 调度的核心概念。

### 1.3 学习目标
- 理解任务调度算法（FIFO、优先级、公平调度）
- 掌握资源管理和隔离技术
- 学会容错和任务重试机制

## 2. 用户画像

### 2.1 目标用户

#### 用户类型 1：HPC 学习者
- **背景**: 计算机科学学生或初级开发者
- **需求**: 学习 HPC 调度系统的基本概念
- **痛点**: 缺乏简单易懂的学习资源
- **期望**: 通过实践理解调度算法和资源管理

#### 用户类型 2：系统开发者
- **背景**: 有经验的系统开发者
- **需求**: 了解调度系统的设计模式
- **痛点**: 现有系统过于复杂，难以理解核心原理
- **期望**: 获得可扩展的调度系统设计参考

#### 用户类型 3：集群管理员
- **背景**: 管理小型 HPC 集群
- **需求**: 简单的任务调度和资源管理
- **痛点**: Slurm 等系统配置复杂
- **期望**: 轻量级、易部署的调度系统

### 2.2 用户场景

#### 场景 1：提交批处理作业
```
用户 -> 提交计算任务 -> 系统分配资源 -> 任务执行 -> 返回结果
```

#### 场景 2：监控任务状态
```
用户 -> 查询任务状态 -> 系统返回状态信息 -> 用户了解进度
```

#### 场景 3：管理集群资源
```
管理员 -> 查看资源使用情况 -> 调整资源分配 -> 优化集群利用率
```

## 3. 功能需求

### 3.1 核心功能

#### F1: 任务提交
- **描述**: 用户可以提交计算任务到调度系统
- **输入**: 任务名称、命令、资源需求、优先级
- **输出**: 任务 ID
- **优先级**: P0（必须实现）

#### F2: 任务查询
- **描述**: 用户可以查询任务状态和结果
- **输入**: 任务 ID
- **输出**: 任务状态、开始时间、完成时间、退出码
- **优先级**: P0

#### F3: 任务取消
- **描述**: 用户可以取消正在运行或等待的任务
- **输入**: 任务 ID
- **输出**: 取消结果
- **优先级**: P0

#### F4: 任务列表
- **描述**: 用户可以查看所有任务列表
- **输入**: 可选的状态过滤
- **输出**: 任务列表
- **优先级**: P0

#### F5: 资源管理
- **描述**: 系统跟踪和管理集群资源
- **输入**: 资源使用情况
- **输出**: 可用资源、已用资源
- **优先级**: P0

#### F6: 调度决策
- **描述**: 系统根据调度算法决定任务执行顺序
- **输入**: 任务队列、可用资源
- **输出**: 调度决策
- **优先级**: P0

### 3.2 扩展功能

#### F7: 任务重试
- **描述**: 任务失败后自动重试
- **输入**: 重试配置
- **输出**: 重试结果
- **优先级**: P1

#### F8: 任务超时
- **描述**: 任务执行超时后自动终止
- **输入**: 超时配置
- **输出**: 超时处理结果
- **优先级**: P1

#### F9: 资源隔离
- **描述**: 使用 cgroups 隔离任务资源
- **输入**: 资源限制配置
- **输出**: 隔离结果
- **优先级**: P2

#### F10: 分布式执行
- **描述**: 任务分发到多个 worker 节点执行
- **输入**: worker 节点配置
- **输出**: 分布式执行结果
- **优先级**: P2

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 任务提交延迟 | < 100ms | 从提交到入队的时间 |
| 调度延迟 | < 1s | 从入队到开始执行的时间 |
| 并发任务数 | > 100 | 系统支持的最大并发任务数 |
| API 响应时间 | < 200ms | HTTP API 的响应时间 |

### 4.2 可靠性需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 可用性 | 99% | 系统正常运行时间 |
| 数据持久性 | 任务状态不丢失 | 重启后恢复状态 |
| 容错性 | 自动重试 | 任务失败后自动重试 |

### 4.3 可扩展性需求

| 维度 | 目标 | 说明 |
|------|------|------|
| 调度算法 | 可插拔 | 支持添加新算法 |
| 资源类型 | 可扩展 | 支持添加新资源类型 |
| 节点数量 | 可扩展 | 支持添加更多节点 |

### 4.4 安全需求

| 需求 | 优先级 | 说明 |
|------|--------|------|
| 输入验证 | P0 | 防止恶意输入 |
| 资源限制 | P0 | 防止资源滥用 |
| 访问控制 | P2 | 用户权限管理 |

## 5. 系统约束

### 5.1 技术约束

- **编程语言**: Go 1.21+
- **操作系统**: Linux（cgroups 支持）
- **依赖**: 尽量减少外部依赖

### 5.2 环境约束

- **部署方式**: 单机部署
- **网络**: 本地网络
- **存储**: 内存存储（不持久化）

### 5.3 时间约束

- **开发周期**: 2-3 天
- **MVP 范围**: 核心功能 F1-F6

## 7. 风险分析

### 7.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 并发 bug | 高 | 中 | 充分测试、使用 race detector |
| 资源泄漏 | 高 | 中 | 严格的资源释放逻辑 |
| 性能瓶颈 | 中 | 低 | 性能测试、优化热点代码 |

### 7.2 进度风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 功能范围蔓延 | 高 | 中 | 严格控制 MVP 范围 |
| 技术难点 | 中 | 中 | 提前调研、寻求帮助 |

## 8. 总结

本项目旨在实现一个简化版的 HPC 任务调度系统，用于学习调度算法和资源管理的核心概念。通过 MVP 版本的实现，可以：

1. 理解任务调度的基本原理
2. 掌握资源管理的技术
3. 学会并发编程的技巧
4. 为后续学习更复杂的系统打下基础
