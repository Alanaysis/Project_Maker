## 1. 核心概念

### 概念1：词法分析（Lexical Analysis）

**定义**：将源代码字符流转换为Token流的过程

**关键点**：
- Token是语言的基本单位（关键字、标识符、数字、运算符等）
- 词法分析器使用有限状态机来识别Token
- 需要处理空白、注释和错误字符

**个人理解**：
词法分析就像阅读文章时识别单词。源代码是一串字符，词法分析器将其分割成有意义的"单词"（Token）。例如 `let x = 5;` 会被识别为 `LET`、`IDENT("x")`、`ASSIGN`、`NUMBER(5)`、`SEMICOL` 五个Token。

### 概念2：语法分析（Parsing）

**定义**：将Token流转换为抽象语法树（AST）的过程

**关键点**：
- 递归下降是一种自顶向下的解析方法
- 运算符优先级决定了表达式的解析顺序
- AST是程序的树形表示

**个人理解**：
语法分析就像理解句子的语法结构。Token流是"单词"，语法分析器将其组织成"句子"（AST）。例如 `1 + 2 * 3` 会被解析为一个树，其中 `*` 的优先级高于 `+`，所以先计算 `2 * 3`。

### 概念3：抽象语法树（AST）

**定义**：程序的树形结构表示，忽略了源代码中的细节（如括号、分号）

**关键点**：
- 每个节点代表一个语法结构
- 语句节点（Statement）和表达式节点（Expression）是两大类
- AST是解释器操作的核心数据结构

**个人理解**：
AST就像文章的提纲。它展示了程序的层次结构，但不包含格式细节。例如 `let x = 5 + 3;` 的AST是：一个LetStatement节点，其Name是Identifier("x")，Value是一个InfixExpression节点，Left是NumberLiteral(5)，Operator是"+"，Right是NumberLiteral(3)。

### 概念4：环境和作用域（Environment & Scope）

**定义**：环境存储变量绑定，作用域决定变量的可见范围

**关键点**：
- 每个环境有一个指向外层环境的引用（作用域链）
- 变量查找沿着作用域链向上进行
- 闭包捕获定义时的环境

**个人理解**：
环境就像嵌套的盒子。每个函数有自己的盒子（环境），盒子里面可以找到变量。如果当前盒子没有，就去外层盒子找。闭包就像函数带着自己的盒子，即使外层函数返回了，盒子还在。

## 2. 重点难点

### 难点1：运算符优先级

**问题**：如何正确解析 `1 + 2 * 3` 为 `1 + (2 * 3)` 而不是 `(1 + 2) * 3`？

**解决过程**：
1. 尝试1：使用简单的递归下降，每个优先级一层函数。问题是代码冗长。
2. 尝试2：使用Pratt解析（优先级爬升）。更优雅。
3. 最终方案：使用Pratt解析，为每个运算符定义优先级。

**关键收获**：
- Pratt解析是一种高效的表达式解析方法
- 优先级表可以清晰地定义运算符的优先级
- 前缀和中缀表达式需要不同的处理方式

### 难点2：闭包实现

**问题**：如何让内部函数访问外部函数的变量？

**解决过程**：
1. 尝试1：复制变量到内部环境。问题是修改不同步。
2. 尝试2：使用指针共享变量。复杂。
3. 最终方案：使用环境链，内部函数捕获定义时的环境。

**关键收获**：
- 闭包的关键是捕获环境，而不是值
- 环境链是实现词法作用域的标准方法
- 函数定义时的环境会被保留，直到函数不再被引用

### 难点3：错误处理

**问题**：如何提供有用的错误信息？

**解决过程**：
1. 尝试1：简单返回error。信息太少。
2. 尝试2：包含行号和列号。更好。
3. 最终方案：在Token中记录位置，在错误信息中包含。

**关键收获**：
- 好的错误信息应该包含位置和上下文
- 词法分析器和语法分析器都应该报告错误
- 运行时错误也需要清晰的错误信息

## 3. 设计决策思考

### 决策1：树遍历 vs 字节码

**背景**：解释器有两种主要实现方式

**我的思考**：
树遍历更简单，适合学习。字节码更快，但实现复杂。作为学习项目，理解原理更重要。

**最终方案**：树遍历

**反思**：
树遍历确实更直观，容易理解执行流程。但如果要优化性能，需要考虑字节码。

### 决策2：递归下降 vs Pratt解析

**背景**：语法分析有多种方法

**我的思考**：
递归下降更直观，但处理运算符优先级需要很多层函数。Pratt解析更优雅，但理解起来稍难。

**最终方案**：混合使用，语句用递归下降，表达式用Pratt解析

**反思**：
这种混合方式很常见，结合了两者的优点。Pratt解析确实让表达式解析更简洁。

### 决策3：Go接口 vs 具体类型

**背景**：如何表示AST节点和运行时对象

**我的思考**：
Go的接口提供了多态能力，但需要类型断言。具体类型更直接，但缺乏灵活性。

**最终方案**：使用接口（Node, Statement, Expression, Object）

**反思**：
接口让代码更灵活，但类型断言增加了复杂度。这是Go的惯用方式，值得学习。

## 4. 代码片段收藏

### 片段1：Pratt解析核心

```go
func (p *Parser) parseExpression(precedence int) Expression {
    // 前缀解析
    leftExp := p.parsePrefix()

    // 中缀解析（优先级爬升）
    for p.peekTok.Type != SEMICOL && precedence < p.peekPrecedence() {
        p.nextToken()
        leftExp = p.parseInfix(leftExp)
    }

    return leftExp
}
```

**为什么收藏**：这是Pratt解析的核心，简洁而优雅

**使用场景**：解析任何需要运算符优先级的表达式

### 片段2：环境链查找

```go
func (e *Environment) Get(name string) (Object, bool) {
    obj, ok := e.store[name]
    if !ok && e.outer != nil {
        obj, ok = e.outer.Get(name)
    }
    return obj, ok
}
```

**为什么收藏**：递归查找实现了词法作用域

**使用场景**：任何需要作用域链的场景

### 片段3：闭包实现

```go
fn := &Function{
    Params: stmt.Params,
    Body:   stmt.Body,
    Env:    interp.env,  // 捕获当前环境
}
```

**为什么收藏**：闭包的关键是捕获环境

**使用场景**：实现任何支持闭包的语言

## 8. 学习心得

### 最大的收获

理解了解释器的完整流程：源代码 -> 词法分析 -> 语法分析 -> AST -> 执行。这个流程不仅适用于脚本语言，也适用于任何需要处理结构化文本的场景。

### 最大的挑战

运算符优先级和闭包是两个主要挑战。Pratt解析解决了优先级问题，环境链解决了闭包问题。

### 给其他学习者的建议

1. 先理解整体架构，再深入细节
2. 写测试！测试是理解代码行为的最好方式
3. 从简单开始，逐步添加功能
4. 不要追求完美，先让它工作
5. 阅读其他人的实现，学习不同的方法
