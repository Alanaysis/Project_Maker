## 2. 重点难点

### 难点1：订单簿数据结构设计

**问题**：如何设计一个高效的数据结构来维护买卖订单？

**解决过程**：
1. 尝试1：使用 `std::vector` 存储所有订单
   - 问题：查找最优价格需要 O(n) 时间
2. 尝试2：使用 `std::map` 按价格分组
   - 优点：可以 O(log n) 获取最优价格
   - 问题：同一价格的多个订单如何维护？
3. 最终方案：使用 `std::map<Price, std::list<Order>>`
   - `std::map` 保证价格有序
   - `std::list` 支持同一价格的 FIFO 顺序

**关键收获**：
- 数据结构的选择要根据使用场景
- 有序容器和无序容器各有优劣
- 组合使用不同容器可以发挥各自优势

### 难点2：撮合算法实现

**问题**：如何实现价格优先、时间优先的撮合逻辑？

**解决过程**：
1. 尝试1：直接遍历所有订单
   - 问题：效率太低
2. 尝试2：只访问最优价格
   - 优点：效率高
   - 问题：如何保证时间优先？
3. 最终方案：使用 `std::list` 的 FIFO 特性
   - 同一价格的订单按添加顺序排列
   - 先添加的订单先成交

**关键收获**：
- 撮合算法的核心是价格优先
- 时间优先通过 FIFO 队列实现
- 成交数量取双方剩余数量的较小值

### 难点3：低延迟优化

**问题**：如何在保证正确性的前提下提高性能？

**解决过程**：
1. 尝试1：使用浮点数存储价格
   - 问题：精度误差，运算慢
2. 尝试2：使用定点数
   - 优点：精度高，运算快
3. 尝试3：优化数据结构
   - 使用 `unordered_map` 实现 O(1) 查找

**关键收获**：
- 金融计算对精度要求极高
- 定点数是更好的选择
- 数据结构的选择直接影响性能

## 4. 代码片段收藏

### 片段1：订单簿数据结构

```cpp
// 买单簿：价格降序排列
std::map<Price, std::list<Order>, std::greater<Price>> bid_book_;

// 卖单簿：价格升序排列
std::map<Price, std::list<Order>, std::less<Price>> ask_book_;
```

**为什么收藏**：这是订单簿的核心数据结构，理解它是理解整个系统的关键

**使用场景**：需要按价格有序存储数据时

```cpp
// 获取最优对手价
```
