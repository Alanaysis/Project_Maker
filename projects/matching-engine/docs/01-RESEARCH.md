# 市场调研报告

## 1. 问题定义

### 要解决的问题

高频交易引擎是金融市场基础设施的核心组件，负责接收买卖订单并按照特定规则进行撮合。本项目旨在实现一个简化但完整的交易引擎，帮助理解：

1. **订单簿的数据结构设计**：如何高效地组织和管理买卖订单
2. **撮合算法的核心逻辑**：如何实现价格优先、时间优先的撮合原则
3. **低延迟系统的优化策略**：如何在保证正确性的前提下提高性能

### 为什么这个问题重要

1. **金融基础设施**：交易引擎是证券交易所、期货交易所的核心系统
2. **高并发挑战**：需要处理每秒数百万笔订单
3. **低延迟要求**：微秒级的响应时间直接影响交易收益
4. **技术深度**：涉及数据结构、算法、系统优化等多个技术领域

## 3. 技术变体分析

### 核心循环的变体

**基础版本**：
```
订单提交 → 订单簿更新 → 撮合匹配 → 成交回报
```

**变体1：价格优先撮合**
```
订单提交 → 检查对手方最优价 → 价格匹配？→ 成交/挂单
```
- 发力方向：价格优先原则
- 为什么这么做：保证公平性，更优价格的订单优先成交
- 适用场景：大多数交易所

**变体2：时间优先撮合**
```
订单提交 → 价格匹配 → 时间排序 → FIFO成交
```
- 发力方向：同一价格下的公平性
- 为什么这么做：先到先得，鼓励流动性提供
- 适用场景：连续竞价市场

**变体3：做市商撮合**
```
订单提交 → 做市商报价 → 价格发现 → 成交
```
- 发力方向：提供流动性
- 为什么这么做：减少买卖价差，提高市场效率
- 适用场景：期权市场、外汇市场

## 4. 技术演进路径

```
[手动交易] → [电子交易] → [算法交易] → [高频交易]
    ↓            ↓            ↓            ↓
 人工下单     系统撮合     自动策略     微秒级延迟
```

### 各阶段特征

1. **手动交易时代**（1990年代以前）
   - 人工在交易大厅喊价
   - 交易速度慢，效率低

2. **电子交易时代**（1990-2000年代）
   - 电子订单簿取代人工喊价
   - 交易速度提升到秒级

3. **算法交易时代**（2000-2010年代）
   - 自动化交易策略
   - 交易速度提升到毫秒级

4. **高频交易时代**（2010年代至今）
   - 微秒级延迟
   - 复杂的市场微观结构策略

## 6. 我们的选择

基于调研，我们选择**价格优先撮合算法**作为核心实现：

### 选择理由

1. **核心概念覆盖**
   - 订单簿数据结构
   - 价格优先、时间优先原则
   - 基本的撮合逻辑

2. **难度适中**
   - 不涉及复杂的市场微观结构
   - 专注于核心算法实现

3. **学习价值高**
   - 是理解更复杂系统的基础
   - 代码清晰，易于理解

4. **可扩展性强**
   - 可以在此基础上添加更多功能
   - 可以优化性能

### 学习价值

- **数据结构选择**：理解为什么选择 `std::map` 而不是 `std::unordered_map`
- **算法设计**：理解撮合的核心逻辑
- **系统设计**：理解低延迟系统的设计原则
- **C++ 高级特性**：模板、迭代器、智能指针等
