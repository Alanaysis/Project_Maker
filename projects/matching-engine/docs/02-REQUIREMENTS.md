# 需求分析文档

## 1. 用户画像

### 主要用户

**用户类型**：C++ 开发者 / 金融科技学习者

**用户特征**：
- 背景：有一定 C++ 基础，想了解金融系统开发
- 技能水平：中级 C++ 开发者
- 学习目标：理解交易引擎的核心原理

**用户痛点**：
1. 不理解订单簿的数据结构设计
2. 不清楚撮合算法的核心逻辑
3. 不知道如何优化系统性能

**用户需求**：
1. 需要一个清晰、可运行的交易引擎示例
2. 需要详细的文档解释核心概念
3. 需要完整的测试用例验证理解

### 次要用户

**用户类型**：面试者

**用户特征**：
- 背景：准备金融科技公司的面试
- 技能水平：有一定编程基础
- 学习目标：能够清晰地解释交易引擎的原理

**用户需求**：
1. 能够快速理解核心概念
2. 能够复述关键设计决策
3. 能够解释性能优化策略

## 2. 使用场景

### 场景1：学习入门

**用户**：C++ 开发者

**目标**：理解交易引擎的基本原理

**步骤**：
1. 阅读 README.md 了解项目概况
2. 编译并运行基础示例
3. 阅读源代码理解核心逻辑
4. 修改代码并观察行为变化

**预期结果**：能够理解订单簿和撮合的基本概念

### 场景2：深入学习

**用户**：金融科技学习者

**目标**：掌握撮合算法的细节

**步骤**：
1. 阅读 03-DESIGN.md 理解设计决策
2. 阅读源代码中的注释
3. 运行测试用例验证理解
4. 完成 LEARNING_NOTES.md 中的练习

**预期结果**：能够独立实现简单的撮合算法

### 场景3：面试准备

**用户**：面试者

**目标**：能够在面试中清晰地解释交易引擎

**步骤**：
1. 阅读 01-RESEARCH.md 了解市场背景
2. 阅读 03-DESIGN.md 理解技术选型
3. 阅读 04-PRODUCT.md 理解产品思维
4. 练习复述关键概念

**预期结果**：能够自信地回答相关面试问题

## 3. 功能需求

### 核心功能（必须）

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F001 | 订单簿管理 | 维护买卖双方的订单簿 | P0 |
| F002 | 限价单支持 | 支持限价买单和卖单 | P0 |
| F003 | 市价单支持 | 支持市价买单和卖单 | P0 |
| F004 | 价格优先撮合 | 按价格优先原则撮合订单 | P0 |
| F005 | 时间优先撮合 | 同一价格按时间优先原则撮合 | P0 |
| F006 | 订单取消 | 支持取消未成交的订单 | P0 |
| F007 | 成交回报 | 撮合成功后返回成交记录 | P0 |

### 扩展功能（可选）

| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F101 | 订单簿深度 | 提供多层级的订单簿深度 | P1 |
| F102 | 订单修改 | 支持修改订单数量 | P1 |
| F103 | 性能统计 | 提供吞吐量和延迟统计 | P2 |
| F104 | 日志记录 | 记录所有操作日志 | P2 |

## 4. 非功能需求

| 需求 | 描述 | 目标 |
|------|------|------|
| 性能 | 单笔订单处理时间 | < 100 微秒 |
| 可用性 | 代码清晰易懂 | 新手能理解核心逻辑 |
| 可维护性 | 模块化设计 | 易于扩展和修改 |
| 可测试性 | 完整的单元测试 | 核心逻辑100%覆盖 |

## 5. 边界定义

### 包含范围

- 订单簿数据结构实现
- 限价单和市价单撮合
- 基本的订单管理
- 单元测试和示例代码

### 不包含范围

- 网络通信层
- 多线程并发处理
- 持久化存储
- 完整的交易系统（如风控、结算）

## 6. 验收标准

- [ ] 能够正确维护订单簿
- [ ] 能够按照价格优先、时间优先原则撮合
- [ ] 能够处理限价单和市价单
- [ ] 能够取消未成交的订单
- [ ] 能够返回正确的成交记录
- [ ] 所有单元测试通过
- [ ] 示例代码能够运行
- [ ] 文档完整且清晰
