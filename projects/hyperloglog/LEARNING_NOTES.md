# HyperLogLog 学习笔记

## 核心概念理解

### 什么是 HyperLogLog?

HyperLogLog (HLL) 是一种概率数据结构，用于估计多集合的基数（不同元素的数量）。它使用极小的内存（通常 1.5KB）就能达到约 2% 的精度。

### 为什么需要 HyperLogLog?

传统方法计算基数需要存储所有元素:
```
问题: 统计网站的独立访客数 (UV)
传统方法: 存储所有访客 ID → 去重 → 计数
内存需求: O(N)，N 为访客数量
```

HyperLogLog 使用概率方法:
```
方法: 观察哈希值的前导零数量
内存: O(m)，m 为寄存器数量 (通常 1024-65536)
精度: 1.04 / √m
```

## 算法原理详解

### 1. 哈希函数

将每个元素映射到均匀分布的哈希值:
```
元素 → 哈希函数 → 64 位哈希值
```

关键点:
- 哈希值必须均匀分布
- 不同元素的哈希值应尽量不同
- 使用 FNV-1a 或 MurmurHash3

### 2. 分桶

使用前 p 位选择寄存器 (桶):
```
哈希值: [b1 b2 ... bp | bp+1 bp+2 ... b64]
         ↑ 分桶索引 (p 位)    ↑ 前导零计数 (64-p 位)
```

关键点:
- p 决定寄存器数量 m = 2^p
- 前 p 位用于分桶
- 剩余位用于前导零计数

### 3. 前导零计数

计算剩余位的前导零数量:
```
剩余位: 000101... → 前导零 = 3
剩余位: 000001... → 前导零 = 4
```

关键点:
- 前导零越多，基数越大
- 使用 CPU 指令计算前导零
- 加 1 避免 log(0) 问题

### 4. 更新寄存器

每个寄存器存储看到的最大前导零数量:
```
register[bucket] = max(register[bucket], leadingZeros)
```

关键点:
- 只能增加，不能减少
- 使用最大值而非平均值
- 支持增量更新

### 5. 基数估算

使用调和平均数计算估计值:
```
E = α * m² * (∑ 2^(-register[i]))^(-1)

其中:
- α: 偏差校正常数
- m: 寄存器数量 (2^p)
- register[i]: 第 i 个寄存器的值
```

关键点:
- 使用调和平均数而非算术平均数
- 小基数使用 Linear Counting 校正
- 大基数使用溢出校正

## 精度分析

### 理论精度

```
标准误差 (SE) = 1.04 / √m

其中 m = 2^p (寄存器数量)
```

### 精度与资源权衡

| 精度 (p) | 寄存器数 | 内存占用 | 标准误差 |
|----------|----------|----------|----------|
| 4        | 16       | 16 字节  | 26%      |
| 8        | 256      | 256 字节 | 6.5%     |
| 10       | 1024     | 1 KB     | 3.25%    |
| 12       | 4096     | 4 KB     | 1.6%     |
| 14       | 16384    | 16 KB    | 0.8%     |
| 16       | 65536    | 64 KB    | 0.4%     |

### 实践建议

- **p=12**: 推荐选择，4KB 内存，1.6% 误差
- **p=14**: 高精度场景，16KB 内存，0.8% 误差
- **p=10**: 资源受限场景，1KB 内存，3.25% 误差

## 实现中的关键问题

### 1. 前导零计数的边界情况

```go
// 问题: 当剩余位全为 0 时
remainingBits := uint64(0)
leadingZeros := bits.LeadingZeros64(remainingBits) // 返回 64

// 解决: 加 1 避免 log(0)
leadingZeros := uint8(bits.LeadingZeros64(remainingBits)) + 1
```

### 2. 分桶索引的位运算

```go
// 问题: 当 p=0 时，右移 64 位
bucketIdx := hash >> (64 - p)  // p=0 时溢出

// 解决: 验证 p >= 4
if p < 4 || p > 16 {
    return nil, ErrInvalidPrecision
}
```

### 3. 浮点精度问题

```go
// 问题: 大基数估算可能溢出
estimate := h.alpha * float64(h.m) * float64(h.m) / sum
// 当 sum 很小时，estimate 可能很大

// 解决: 使用大基数校正
if estimate > 143165576.5 { // 2^32 / 30
    estimate = -math.Pow(2, 32) * math.Log(1-estimate/math.Pow(2, 32))
}
```

### 4. 调和平均数 vs 算术平均数

```go
// 算术平均数: (a + b + c) / 3
// 调和平均数: 3 / (1/a + 1/b + 1/c)

// 调和平均数的优势:
// - 对离群值不敏感
// - 更适合倒数关系的数据
// - HyperLogLog 的估计基于 2^(-register)
```

## 校正策略

### 小基数校正 (Linear Counting)

```
当 estimate <= 2.5 * m 时:
E = m * ln(m / V)
其中 V = 空寄存器数量
```

原因:
- 小基数时，调和平均数估算不准确
- Linear Counting 对小基数更准确
- 阈值 2.5 * m 来自论文建议

### 大基数校正

```
当 estimate > 2^32 / 30 时:
E = -2^32 * ln(1 - E/2^32)
```

原因:
- 大基数时，浮点溢出风险
- 校正公式避免溢出
- 阈值 2^32 / 30 来自论文建议

## 合并操作

### 合并原理

```go
func (h *HyperLogLog) Merge(other *HyperLogLog) error {
    // 检查精度是否相同
    if h.p != other.p {
        return fmt.Errorf("precision mismatch")
    }

    // 取每个寄存器的最大值
    for i := uint32(0); i < h.m; i++ {
        if other.registers[i] > h.registers[i] {
            h.registers[i] = other.registers[i]
        }
    }
    return nil
}
```

关键点:
- 要求精度相同
- 逐寄存器取最大值
- 合并是幂等的

### 应用场景

```
分布式场景:
1. 多个节点各自统计
2. 定期合并结果
3. 获得全局估算

示例:
节点 A: 统计用户 1-1000
节点 B: 统计用户 500-1500
合并后: 估算用户 1-1500 的基数
```


## 与其他算法比较

### Linear Counting

```
适用: 小基数 (m/V > 2.5)
精度: 高
内存: O(m)
公式: E = m * ln(m/V)
```

### LogLog

```
适用: 中等基数
精度: 中等 (1.30/√m)
内存: O(m)
改进: HyperLogLog 使用调和平均数
```

### HyperLogLog

```
适用: 大基数
精度: 高 (1.04/√m)
内存: O(m)
优势: 更好的偏差校正
```

## 性能考虑

### 添加性能

```
时间复杂度: O(1)
空间复杂度: O(m)
操作: 哈希 + 位运算 + 比较
```

### 估算性能

```
时间复杂度: O(m)
空间复杂度: O(1)
操作: 遍历寄存器 + 浮点运算
```

### 合并性能

```
时间复杂度: O(m)
空间复杂度: O(1)
操作: 逐寄存器比较
```
