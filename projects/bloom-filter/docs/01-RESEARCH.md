# 布隆过滤器研究文档

## 1. 概述

布隆过滤器 (Bloom Filter) 是一种空间效率极高的概率型数据结构，由 Burton Howard Bloom 于 1970 年提出。它用于高效地判断一个元素是否属于一个集合。

### 核心特性

- **空间效率**: 使用位数组存储，远小于哈希表
- **查询效率**: O(k)，k 为哈希函数数量
- **插入效率**: O(k)
- **假阳性**: 可能将不存在的元素误判为存在
- **假阴性**: 绝不会将存在的元素误判为不存在

### 应用场景

1. **网络爬虫**: URL 去重
2. **垃圾邮件过滤**: 关键词匹配
3. **数据库优化**: 减少不必要的磁盘查询
4. **缓存系统**: 防止缓存穿透
5. **分布式系统**: 成员查询

## 2. 数学原理

### 2.1 基本结构

布隆过滤器由两部分组成:
1. 一个长度为 m 的位数组 (bit array)，初始值全为 0
2. k 个独立的哈希函数 h1, h2, ..., hk

### 2.2 插入操作

插入元素 x:
1. 计算 k 个哈希值: h1(x), h2(x), ..., hk(x)
2. 将位数组中对应位置设为 1

```
bits[h1(x)] = 1
bits[h2(x)] = 1
...
bits[hk(x)] = 1
```

### 2.3 查询操作

查询元素 x:
1. 计算 k 个哈希值: h1(x), h2(x), ..., hk(x)
2. 检查所有对应位置是否都为 1

- 如果任何位置为 0 -> 元素一定不存在
- 如果所有位置都为 1 -> 元素可能存在

### 2.4 误判率

假阳性概率的精确公式:

```
p = (1 - (1 - 1/m)^(kn))^k
```

当 m 和 n 很大时，近似为:

```
p ~ (1 - e^(-kn/m))^k
```

其中:
- p = 假阳性概率
- k = 哈希函数数量
- n = 已插入元素数量
- m = 位数组大小

### 2.5 最优参数

给定期望的假阳性率 p 和预期元素数量 n，最优参数为:

**最优位数组大小:**
```
m = -(n * ln(p)) / (ln(2))^2
```

**最优哈希函数数量:**
```
k = (m / n) * ln(2)
```

**每元素比特数:**
```
bits_per_element = -ln(p) / (ln(2))^2 ~ -1.44 * log2(p)
```

## 3. 变体

### 3.1 计数布隆过滤器 (Counting Bloom Filter)

由 Fan 等人于 2000 年提出，使用计数器替代位数组，支持删除操作。

**特点:**
- 每个位置使用 8 位计数器
- 支持插入、查询和删除
- 空间开销约为标准布隆过滤器的 8 倍
- 删除操作存在假阴性风险

**删除操作:**
```
remove(x):
  for i in 1..k:
    if counters[hi(x)] == 0:
      return false  // 元素不存在
  for i in 1..k:
    counters[hi(x)] -= 1
  return true
```

### 3.2 可扩展布隆过滤器 (Scalable Bloom Filter)

由 Almeida 等人于 2007 年提出，支持动态扩容。

**特点:**
- 当误判率超过阈值时自动创建新过滤器
- 每层的误判率递减，保证总体误判率不超过目标
- 空间动态增长

**扩容策略:**
1. 初始过滤器使用目标误判率 p
2. 每次扩容，新过滤器的误判率为 p * r^i (r < 1)
3. 总体误判率不超过 p / (1 - r)

### 3.3 压缩布隆过滤器 (Compressed Bloom Filter)

通过压缩位数组减少空间使用，适用于网络传输场景。

### 3.4 稳健布隆过滤器 (Robust Bloom Filter)

对哈希函数攻击具有抵抗力，使用密码学哈希函数。

## 4. 哈希函数

### 4.1 双重哈希技术

使用双重哈希技术从两个基础哈希函数生成 k 个哈希值:

```
hi(x) = (h1(x) + i * h2(x)) mod m
```

**优势:**
1. 只需计算两个哈希值
2. 计算效率高
3. 哈希值之间具有良好的独立性

### 4.2 常用哈希函数

1. **MurmurHash**: 非密码学哈希，分布均匀，计算快速
2. **CityHash**: Google 开发，适合字符串哈希
3. **FNV-1a**: 简单高效，适合小数据
4. **SHA-256**: 密码学哈希，安全性高但计算慢

### 4.3 哈希函数选择

| 场景 | 推荐哈希函数 |
|------|-------------|
| 一般用途 | MurmurHash3 |
| 字符串处理 | CityHash, xxHash |
| 安全敏感 | SHA-256 |
| 嵌入式系统 | FNV-1a |

## 5. 性能分析

### 5.1 时间复杂度

| 操作 | 标准布隆过滤器 | 计数布隆过滤器 | 可扩展布隆过滤器 |
|------|--------------|--------------|----------------|
| 插入 | O(k) | O(k) | O(k * L) |
| 查询 | O(k) | O(k) | O(k * L) |
| 删除 | N/A | O(k) | N/A |

k = 哈希函数数量，L = 过滤器层数

### 5.2 空间复杂度

| 类型 | 空间复杂度 | 每元素字节 (p=0.01) |
|------|----------|-------------------|
| 标准布隆过滤器 | O(m/8) | 9.6 字节 |
| 计数布隆过滤器 | O(m) | 76.8 字节 |
| 可扩展布隆过滤器 | 动态增长 | ~12 字节 |

### 5.3 误判率对比

| 期望误判率 | 每元素比特数 | 哈希函数数量 |
|-----------|------------|------------|
| 0.1 | 4.8 | 3 |
| 0.01 | 9.6 | 7 |
| 0.001 | 14.4 | 10 |
| 0.0001 | 19.2 | 13 |

## 6. 实际应用案例

### 6.1 Google Chrome

使用布隆过滤器进行恶意 URL 检测。

### 6.2 Apache Cassandra

使用布隆过滤器减少不必要的磁盘查询。

### 6.3 Redis

使用布隆过滤器模块进行成员查询。

### 6.4 Bitcoin

使用布隆过滤器进行轻量级钱包交易过滤。

## 7. 局限性

1. **假阳性**: 无法 100% 确认元素存在
2. **不支持删除**: 标准布隆过滤器不支持删除操作
3. **容量固定**: 标准布隆过滤器容量固定，需要预估元素数量
4. **哈希函数依赖**: 性能依赖于哈希函数的质量
