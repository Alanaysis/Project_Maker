# 分布式缓存研究文档

## 1. 缓存基础概念

### 1.1 什么是缓存

缓存是一种存储技术，将频繁访问的数据存储在快速访问的存储介质中，以减少对慢速存储的访问次数。

**核心价值**：
- 降低延迟：内存访问比磁盘快 10^5 倍
- 提高吞吐：减少后端负载
- 降低成本：减少数据库/网络开销

### 1.2 缓存层次

```
CPU Cache (L1/L2/L3) → 应用缓存 → 分布式缓存 → 数据库
    ↓                    ↓            ↓           ↓
  纳秒级               微秒级        毫秒级       毫秒-秒级
```

### 1.3 缓存命中率

缓存命中率是衡量缓存效果的核心指标：

```
命中率 = 缓存命中次数 / (缓存命中次数 + 缓存未命中次数)
```

**理想命中率**: > 90%

## 2. 缓存淘汰策略

### 2.1 LRU (Least Recently Used)

**原理**: 淘汰最久未被访问的数据

**实现**: 双向链表 + HashMap
- 访问时：将节点移到链表头部
- 淘汰时：删除链表尾部节点

**时间复杂度**: O(1)

**适用场景**: 访问模式具有时间局部性

**优缺点**:
- 优点：实现简单，效果好
- 缺点：扫描攻击会污染缓存

### 2.2 LFU (Least Frequently Used)

**原理**: 淘汰访问频率最低的数据

**实现**: 频率桶
- 每个频率对应一个桶
- 访问时：从旧频率桶移到新频率桶
- 淘汰时：从最低频率桶中淘汰

**时间复杂度**: O(1)

**适用场景**: 访问模式具有频率特征

**优缺点**:
- 优点：更精确反映数据热度
- 缺点：新数据容易被淘汰（频率为1）

### 2.3 FIFO (First In First Out)

**原理**: 淘汰最早进入缓存的数据

**实现**: 队列
- 入缓存时：加入队列尾部
- 淘汰时：从队列头部删除

**时间复杂度**: O(1)

**适用场景**: 数据具有时效性，新数据更重要

**优缺点**:
- 优点：实现最简单
- 缺点：不考虑访问模式，效果一般

### 2.4 TTL (Time To Live)

**原理**: 基于过期时间淘汰

**实现**: 最小堆
- 每个数据设置过期时间
- 定期检查并删除过期数据

**时间复杂度**: O(log n)

**适用场景**: 数据具有明确的生命周期

**优缺点**:
- 优点：精确控制数据生命周期
- 缺点：需要定期清理

### 2.5 策略对比

| 策略 | 时间复杂度 | 空间复杂度 | 适用场景 | 实现难度 |
|------|-----------|-----------|----------|----------|
| LRU | O(1) | O(n) | 时间局部性 | 简单 |
| LFU | O(1) | O(n) | 频率特征 | 中等 |
| FIFO | O(1) | O(n) | 时效性数据 | 简单 |
| TTL | O(log n) | O(n) | 明确过期时间 | 中等 |

## 3. 一致性哈希

### 3.1 传统哈希的问题

传统哈希: `node = hash(key) % N`

问题：当 N 变化时，几乎所有数据都需要重新分配

### 3.2 一致性哈希原理

**核心思想**: 将哈希值空间组织成一个环（0 ~ 2^32-1）

```
        0
       / \
      /   \
   node1   node2
      \   /
       \ /
      node3
```

**映射规则**:
1. 节点和数据都映射到环上
2. 数据顺时针找到第一个节点
3. 节点变化时只影响相邻数据

### 3.3 虚拟节点

**问题**: 节点少时数据分布不均匀

**解决**: 每个物理节点对应多个虚拟节点

```
物理节点 A → 虚拟节点 A#1, A#2, A#3, ...
物理节点 B → 虚拟节点 B#1, B#2, B#3, ...
```

**效果**: 数据分布更均匀

### 3.4 节点扩缩容

**扩容**:
1. 新节点加入环
2. 从相邻节点获取数据
3. 只影响部分数据

**缩容**:
1. 节点离开环
2. 数据迁移到相邻节点
3. 只影响部分数据

**数据迁移量**: 理论上为 K/N（K=总数据量，N=节点数）

## 4. 缓存模式

### 4.1 Cache-Aside

**流程**:
```
读: 应用 → 检查缓存 → 命中? 返回 : 查数据库 → 存缓存 → 返回
写: 应用 → 更新数据库 → 删除缓存
```

**特点**:
- 应用负责缓存管理
- 缓存和数据库可能不一致
- 实现最灵活

**适用**: 大多数场景

### 4.2 Read-Through

**流程**:
```
读: 应用 → 缓存 → 命中? 返回 : 缓存自动加载 → 返回
```

**特点**:
- 缓存自动加载数据
- 应用代码更简洁
- 缓存层更复杂

**适用**: 读多写少场景

### 4.3 Write-Through

**流程**:
```
写: 应用 → 缓存 → 同步写入数据库 → 返回
```

**特点**:
- 数据强一致性
- 写延迟较高
- 实现较复杂

**适用**: 对一致性要求高的场景

### 4.4 Write-Behind

**流程**:
```
写: 应用 → 缓存 → 返回
                → 异步写入数据库
```

**特点**:
- 写延迟最低
- 可能丢失数据
- 实现最复杂

**适用**: 写密集型场景

### 4.5 模式对比

| 模式 | 一致性 | 延迟 | 复杂度 | 适用场景 |
|------|--------|------|--------|----------|
| Cache-Aside | 最终一致 | 中 | 低 | 通用 |
| Read-Through | 最终一致 | 中 | 中 | 读多写少 |
| Write-Through | 强一致 | 高 | 中 | 高一致性 |
| Write-Behind | 最终一致 | 低 | 高 | 写密集型 |

## 5. 缓存问题

### 5.1 缓存穿透

**定义**: 查询不存在的数据，缓存永远无法命中

**攻击方式**: 大量请求不存在的键

**解决方案**:
1. **布隆过滤器**: 预先判断键是否存在
2. **空值缓存**: 缓存不存在的键，设置较短 TTL
3. **接口校验**: 在接口层过滤非法请求

### 5.2 缓存击穿

**定义**: 热点键过期，大量请求同时访问数据库

**场景**: 秒杀、热门商品

**解决方案**:
1. **Single Flight**: 合并并发请求
2. **互斥锁**: 使用 double-check 机制
3. **永不过期**: 逻辑过期 + 后台更新

### 5.3 缓存雪崩

**定义**: 大量缓存同时过期，请求全部打到数据库

**场景**: 缓存服务器重启、大量键同时设置 TTL

**解决方案**:
1. **随机 TTL**: 避免同时过期
2. **多级缓存**: L1 + L2
3. **限流降级**: 保护后端服务
4. **高可用**: 缓存集群

### 5.4 问题对比

| 问题 | 原因 | 影响 | 解决方案 |
|------|------|------|----------|
| 穿透 | 键不存在 | 数据库压力 | 布隆过滤器 |
| 击穿 | 热点键过期 | 数据库压力 | Single Flight |
| 雪崩 | 大量键过期 | 数据库崩溃 | 随机TTL + 多级缓存 |

## 6. 分布式特性

### 6.1 节点发现

**方式**:
1. **静态配置**: 预先配置节点列表
2. **注册中心**: ZooKeeper、etcd、Consul
3. **Gossip 协议**: 去中心化节点发现

**选择**:
- 小规模：静态配置
- 中规模：注册中心
- 大规模：Gossip 协议

### 6.2 数据复制

**策略**:
1. **主从复制**: 一主多从，读写分离
2. **多主复制**: 多个主节点，冲突解决
3. **无主复制**: 客户端直接访问多个节点

**一致性级别**:
- **ONE**: 写入一个节点即返回
- **QUORUM**: 写入多数节点即返回
- **ALL**: 写入所有节点才返回

### 6.3 故障转移

**检测**:
- 心跳检测（每 N 秒）
- 超时判定
- 多次确认

**处理**:
1. 标记节点为故障
2. 将请求路由到其他节点
3. 触发数据恢复
4. 节点恢复后重新加入

**重试策略**:
- 指数退避
- 最大重试次数
- 熔断机制

## 7. 实际应用场景

### 7.1 热点数据缓存

**场景**: 电商商品详情、新闻热点

**策略**:
- 热点检测：访问计数
- 动态 TTL：热点数据延长过期时间
- 多级缓存：L1 (本地) + L2 (分布式)

### 7.2 会话存储

**场景**: 用户登录状态、购物车

**特点**:
- 临时性：有过期时间
- 私有性：每个用户独立
- 频繁访问：每次请求都需要

**实现**:
- Session ID 作为键
- 设置合理 TTL
- 定期清理过期会话

### 7.3 限流器

**场景**: API 限流、防刷

**算法**:
1. **固定窗口**: 简单但有边界问题
2. **滑动窗口**: 更平滑
3. **令牌桶**: 允许突发流量
4. **漏桶**: 恒定速率

**实现**:
- 使用缓存存储计数器
- 设置窗口 TTL
- 原子操作保证并发安全

## 8. 性能优化

### 8.1 缓存预热

**目的**: 避免冷启动

**方式**:
- 批量加载热点数据
- 后台定时更新
- 懒加载 + 预加载结合

### 8.2 缓存降级

**目的**: 保护后端服务

**策略**:
- 返回默认值
- 返回历史数据
- 直接拒绝请求

### 8.3 缓存监控

**指标**:
- 命中率
- 延迟
- 内存使用
- 连接数

**工具**:
- Prometheus + Grafana
- 自定义监控
