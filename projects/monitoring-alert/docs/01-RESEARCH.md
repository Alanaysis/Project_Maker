# 01 - 市场调研：监控告警系统

## 1. 监控系统概述

### 1.1 什么是监控系统

监控系统是用于收集、存储、分析和可视化系统运行状态的软件系统。它帮助运维人员：

- 实时了解系统健康状态
- 快速发现和定位问题
- 预测潜在风险
- 优化系统性能

### 1.2 监控系统的核心组件

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   指标采集   │ -> │   数据存储   │ -> │   查询分析   │ -> │   告警通知   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
     │                   │                   │                   │
     v                   v                   v                   v
  采集器             时序数据库           查询引擎           通知器
```

### 1.3 监控指标类型

| 类型 | 描述 | 示例 |
|------|------|------|
| Counter | 只增不减的计数器 | 请求数、错误数、字节数 |
| Gauge | 可增可减的仪表值 | 温度、内存使用率、队列长度 |
| Histogram | 统计数据分布 | 请求延迟分布、响应大小分布 |
| Summary | 类似 Histogram，计算分位数 | P99 延迟、P95 延迟 |

## 2. 现有实现分析

### 2.1 主流监控系统

| 系统 | 特点 | 适用场景 |
|------|------|----------|
| Prometheus | 拉取模型、强大的查询语言 | 云原生、微服务 |
| Graphite | 推送模型、简单易用 | 传统运维 |
| InfluxDB | 专为时序数据设计 | IoT、实时分析 |
| Zabbix | 企业级、功能全面 | 传统企业 |

| 数据库 | 语言 | 特点 | 性能 |
|--------|------|------|------|
| Prometheus TSDB | Go | 嵌入式、高效压缩 | 高 |

| OpenTSDB | Java | HBase 存储、分布式 | 中 |

## 3. 时序数据存储技术

### 3.1 时序数据特点

- **时间戳主导**：数据按时间顺序排列
- **写入密集**：持续高频写入
- **查询模式**：按时间范围查询为主
- **数据压缩**：时间戳和值都可以高效压缩

### 3.2 存储结构

```
┌─────────────────────────────────────────────┐
│                  时序数据块                   │
├─────────────────────────────────────────────┤
│  时间戳序列  │  值序列  │  标签索引  │  元数据  │
└─────────────────────────────────────────────┘
```

### 3.3 压缩算法

- **时间戳压缩**：Delta-of-delta 编码
- **值压缩**：XOR 编码（针对浮点数）
- **标签压缩**：字典编码

## 4. 告警规则引擎

### 4.1 告警规则结构

```yaml
groups:
  - name: example
    rules:
      - alert: HighCPU
        expr: cpu_usage > 80
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "CPU usage is high"
```

### 4.2 规则评估流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  解析规则   │ -> │  查询数据   │ -> │  评估条件   │ -> │  触发告警   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### 4.3 告警状态机

```
                 ┌─────────────┐
                 │   Pending   │
                 └──────┬──────┘
                        │ 持续触发
                        v
                 ┌─────────────┐
                 │   Firing    │
                 └──────┬──────┘
                        │ 条件不满足
                        v
                 ┌─────────────┐
                 │  Resolved   │
                 └─────────────┘
```

## 5. 通知机制

### 5.1 通知方式

| 方式 | 优点 | 缺点 |
|------|------|------|
| 日志 | 简单、无依赖 | 不实时 |
| Webhook | 灵活、可集成 | 需要接收端 |
| 邮件 | 通用、可靠 | 延迟高 |
| Slack | 实时、协作 | 依赖外部服务 |

### 5.2 通知策略

- **即时通知**：告警触发后立即发送
- **节流通知**：限制通知频率，避免告警风暴
- **分组通知**：将相关告警合并发送
- **静默规则**：在特定时间段抑制通知
