# 需求文档: 时间序列数据库

## 1. 功能需求

### 1.1 数据写入

#### FR-W01: 单点写入
- 支持写入单个数据点
- 数据点包含: metric, tags, timestamp, value
- 返回写入状态

#### FR-W02: 批量写入
- 支持批量写入多个数据点
- 支持同一 metric 的批量写入
- 支持不同 metric 的混合批量写入
- 批量大小: 1-10000 points

#### FR-W03: 时间戳处理
- 支持秒级和毫秒级时间戳
- 自动排序 (按时间戳升序)
- 支持自动填充当前时间

#### FR-W04: 数据验证
- metric 名称非空
- timestamp 为有效时间戳
- value 为数值类型 (int/float)
- tags 为键值对

### 1.2 数据查询

#### FR-Q01: 时间范围查询
```sql
SELECT * FROM metric
WHERE timestamp >= start AND timestamp <= end
```
- 支持指定起止时间
- 返回时间范围内的所有数据点
- 结果按时间戳排序

#### FR-Q02: 标签过滤
```sql
SELECT * FROM metric
WHERE tags['host'] = 'server1'
```
- 支持单标签过滤
- 支持多标签 AND 过滤
- 支持标签存在性检查

#### FR-Q03: 聚合查询
支持以下聚合函数:
- `avg`: 平均值
- `max`: 最大值
- `min`: 最小值
- `sum`: 求和
- `count`: 计数
- `first`: 第一个值
- `last`: 最后一个值
- `stddev`: 标准差

#### FR-Q04: 降采样
```sql
SELECT avg(value) FROM metric
WHERE timestamp >= start AND timestamp <= end
GROUP BY time(60s)
```
- 支持按时间间隔降采样
- 支持指定聚合函数
- 支持填充策略 (null, 0, 前值)

#### FR-Q05: 最新值查询
- 查询指定 metric 的最新数据点
- 支持标签过滤
- 响应时间 < 1ms

### 1.3 存储引擎

#### FR-S01: 内存表 (MemTable)
- 使用有序数据结构 (跳表/红黑树)
- 支持快速插入和查询
- 容量限制: 可配置 (默认 64MB)
- 达到容量后触发刷盘

#### FR-S02: WAL (Write-Ahead Log)
- 写入前先写 WAL
- 顺序写入，高性能
- 支持 WAL 恢复
- WAL 文件轮转

#### FR-S03: 持久化存储
- MemTable 刷盘为 SSTable 文件
- 支持数据压缩
- 支持数据校验 (CRC32)
- 文件格式自描述

#### FR-S04: 数据压缩
- 时间戳压缩: Delta + Simple8b
- 值压缩: Gorilla 编码
- 标签压缩: 字典编码
- 压缩率目标: > 10:1

### 1.4 数据保留

#### FR-R01: TTL 过期
- 支持全局默认 TTL
- 支持 metric 级别 TTL
- TTL 单位: 秒

#### FR-R02: 自动清理
- 后台定期检查过期数据
- 支持配置检查间隔
- 支持手动触发清理

#### FR-R03: 数据保留策略
- 支持按时间保留 (保留最近 N 天)
- 支持按大小保留 (保留最近 N GB)
- 支持按 metric 配置不同策略

### 1.5 API 接口

#### FR-A01: 写入 API
```
POST /write
Content-Type: application/json

{
    "metric": "cpu_usage",
    "tags": {"host": "server1"},
    "points": [
        {"timestamp": 1625097600, "value": 45.2}
    ]
}
```

#### FR-A02: 查询 API
```
GET /query?metric=cpu_usage&start=1625097600&end=1625098600&agg=avg&downsample=60
```

#### FR-A03: 批量查询 API
```
POST /query/batch
Content-Type: application/json

{
    "queries": [
        {"metric": "cpu_usage", "start": 1625097600, "end": 1625098600},
        {"metric": "memory_usage", "start": 1625097600, "end": 1625098600}
    ]
}
```

#### FR-A04: 健康检查 API
```
GET /health
```

## 2. 非功能需求

### 2.1 性能需求

#### NFR-P01: 写入性能
- 单点写入: > 10,000 points/sec
- 批量写入: > 100,000 points/sec
- 写入延迟: < 1ms (P99)

#### NFR-P02: 查询性能
- 时间范围查询: < 10ms (1M points)
- 聚合查询: < 50ms (1M points)
- 降采样查询: < 100ms (10M points)
- 最新值查询: < 1ms

#### NFR-P03: 存储效率
- 压缩率: > 10:1
- 磁盘空间利用率: > 80%

### 2.2 可靠性需求

#### NFR-R01: 数据持久性
- WAL 保证写入不丢失
- 支持崩溃恢复
- 数据校验防止损坏

#### NFR-R02: 数据一致性
- 写入后立即可读
- 查询结果一致性

### 2.3 可扩展性需求

#### NFR-S01: 数据容量
- 支持单机 100GB+ 数据
- 支持 1000+ metric
- 支持 10000+ 标签组合

#### NFR-S02: 并发支持
- 支持多写入并发
- 支持多查询并发
- 支持读写并发

### 2.4 可维护性需求

#### NFR-M01: 日志
- 操作日志
- 错误日志
- 性能日志

#### NFR-M02: 监控指标
- 写入 QPS
- 查询 QPS
- 存储使用量
- 内存使用量

## 3. 约束条件

### 3.1 技术约束
- 使用 Python 3.10+
- 无外部数据库依赖
- 文件系统存储

### 3.2 资源约束
- 内存使用: < 1GB (默认配置)
- CPU 使用: < 50% (单核)
- 磁盘 I/O: 顺序写优先

### 3.3 兼容性约束
- 支持 Linux/macOS
- 支持本地文件系统
- 支持 SSD/HDD

## 4. 用例场景

### 4.1 监控数据存储

**场景描述**:
- 100 台服务器，每台 10 个指标
- 每 10 秒采集一次
- 保留 30 天

**数据量计算**:
- 每秒数据点: 100 * 10 / 10 = 100 points/sec
- 每天数据点: 100 * 86400 = 8.64M points
- 30 天数据点: 259.2M points
- 存储空间 (压缩后): ~2.5GB

### 4.2 IoT 数据存储

**场景描述**:
- 1000 个传感器，每个 5 个指标
- 每分钟采集一次
- 保留 90 天

**数据量计算**:
- 每秒数据点: 1000 * 5 / 60 = 83 points/sec
- 每天数据点: 1000 * 5 * 1440 = 7.2M points
- 90 天数据点: 648M points
- 存储空间 (压缩后): ~6.5GB
