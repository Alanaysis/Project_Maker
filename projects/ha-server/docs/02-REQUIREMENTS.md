# 02 - 需求分析

## 一、项目目标

构建一个教学级的高可用服务器框架，帮助学习者理解：
- 高可用架构的核心设计原理
- 负载均衡算法的实现方式
- 故障检测和转移机制
- 高性能网络编程技术

## 三、功能需求

### 3.1 核心功能（P0 - 必须实现）

#### F1: 请求代理
- 接收客户端 HTTP 请求
- 解析请求头和请求体
- 转发请求到后端服务器
- 返回响应给客户端

#### F2: 负载均衡
- **轮询 (Round Robin)**：按顺序分配请求到各后端
- **加权轮询 (Weighted Round Robin)**：根据权重分配请求
- **最少连接 (Least Connections)**：分配到连接数最少的后端

#### F3: 健康检查
- **主动检查**：定期向后端发送探测请求
- **被动检查**：根据请求响应判断后端状态
- **检查间隔**：可配置的检查频率
- **失败阈值**：连续失败次数达到阈值后标记为不可用

#### F4: 故障转移
- 自动跳过不可用的后端
- 后端恢复后自动重新加入
- 所有后端不可用时返回错误响应

#### F5: 连接池管理
- 预建立到后端的连接
- 连接复用，减少握手开销
- 连接超时回收
- 连接池大小可配置

### 3.2 增强功能（P1 - 建议实现）

#### F6: 配置管理
- JSON 格式配置文件
- 后端服务器列表配置
- 负载均衡策略配置
- 健康检查参数配置

#### F7: 日志记录
- 请求日志（访问日志）
- 错误日志
- 健康检查日志
- 日志级别可配置

#### F8: 统计信息
- 请求总数和 QPS
- 各后端请求数
- 错误率统计
- 响应时间统计

### 3.3 扩展功能（P2 - 可选实现）

#### F9: 会话保持
- 基于 IP 的会话保持
- 基于 Cookie 的会话保持

#### F10: 限流
- 基于 IP 的限流
- 全局限流
- 令牌桶算法

## 四、非功能需求

### 4.1 性能要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 并发连接数 | 1000+ | 单实例支持的并发连接 |
| 请求延迟 | < 10ms | 代理转发的额外延迟 |
| QPS | 10000+ | 每秒处理请求数 |
| 内存占用 | < 100MB | 空闲状态内存占用 |

### 4.2 可用性要求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 故障检测时间 | < 5s | 从后端故障到检测到的时间 |
| 故障转移时间 | < 1s | 从检测到故障到切换完成的时间 |
| 恢复检测时间 | < 10s | 从后端恢复到重新加入的时间 |

### 4.3 可靠性要求

- 不丢失已建立的连接（在后端切换时）
- 配置错误时不崩溃，返回错误信息
- 优雅处理各种异常情况

## 五、技术约束

### 5.1 平台要求
- 操作系统：Linux（使用 epoll）
- 编译器：GCC 9+ 或 Clang 10+
- C++ 标准：C++17

### 5.2 依赖要求
- 无第三方库依赖（从零实现）
- 仅使用系统标准库

### 5.3 接口约束
- HTTP/1.1 协议支持
- 支持 GET、POST 等基本方法
- 支持 Keep-Alive 连接
