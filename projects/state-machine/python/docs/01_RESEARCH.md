# 状态机框架研究

## 1. 概述

状态机（State Machine）是一种计算模型，用于描述对象在其生命周期内所经历的状态序列。状态机框架是实现状态机的软件库，提供定义状态、转换和事件的标准化方式。

## 2. 状态机类型

### 2.1 有限状态机（FSM）

有限状态机是最简单的状态机类型，包含：
- **有限的状态集合**：系统可以处于的有限个状态
- **转换规则**：定义状态之间如何转换
- **事件**：触发状态转换的事件
- **当前状态**：系统当前所处的状态

**应用场景**：
- 交通灯控制
- 简单的工作流
- 游戏中的简单AI

### 2.2 层次状态机（HSM）

层次状态机支持嵌套状态，允许将复杂状态分解为更小的子状态：
- **嵌套状态**：状态可以包含子状态
- **历史状态**：记住上次离开时的子状态
- **事件冒泡**：事件可以向上传播到父状态

**应用场景**：
- 复杂的游戏AI
- 用户界面状态管理
- 协议实现

### 2.3 并发状态机

并发状态机允许多个状态机同时运行：
- **并行状态**：多个状态同时活动
- **同步机制**：状态机之间的同步

**应用场景**：
- 多线程系统
- 复杂的业务流程

## 3. 核心概念

### 3.1 状态（State）

状态表示系统在某个时刻的条件或情况：
- **命名**：每个状态都有唯一的名称
- **元数据**：可以附加额外数据
- **动作**：进入和退出时执行的操作

### 3.2 事件（Event）

事件是触发状态转换的信号：
- **命名**：每个事件都有唯一的名称
- **数据**：可以携带额外数据
- **不可变**：事件一旦创建不应修改

### 3.3 转换（Transition）

转换定义状态之间的移动：
- **源状态**：转换开始的状态
- **目标状态**：转换结束的状态
- **触发事件**：引发转换的事件
- **守卫条件**：转换必须满足的条件
- **动作**：转换时执行的操作

### 3.4 守卫条件（Guard）

守卫条件是布尔表达式，决定转换是否允许发生：
- **条件检查**：基于当前状态、事件和上下文
- **组合逻辑**：支持AND、OR、NOT操作
- **延迟求值**：在转换时才计算

### 3.5 动作（Action）

动作是在转换过程中执行的操作：
- **进入动作**：进入状态时执行
- **退出动作**：离开状态时执行
- **转换动作**：转换过程中执行

## 4. 设计模式

### 4.1 建造者模式（Builder Pattern）

用于逐步构建复杂的转换：

```python
transition = (TransitionBuilder()
    .from_state(State("Idle"))
    .to_state(State("Active"))
    .on(Event("Start"))
    .when(guard_condition)
    .do(action_function)
    .describe("Start the process")
    .build())
```

### 4.2 观察者模式（Observer Pattern）

用于监听状态变化：

```python
sm.on_state_change(lambda from_s, to_s, event: 
    print(f"State changed: {from_s} -> {to_s}"))
```

### 4.3 策略模式（Strategy Pattern）

用于动态改变守卫和动作：

```python
guard = FunctionGuard(lambda f, t, e, c: c.get("authorized", False))
```

## 5. 实际应用

### 5.1 订单状态管理

订单生命周期：
- 创建 -> 已验证 -> 待支付 -> 已支付 -> 配送中 -> 已发货 -> 已完成
- 可以取消、退款等

### 5.2 工作流引擎

文档审批流程：
- 草稿 -> 审核中 -> 已批准 -> 已发布
- 可以拒绝、修改、归档等

### 5.3 游戏AI

游戏NPC行为：
- 巡逻 -> 发现敌人 -> 追击 -> 攻击 -> 逃跑
- 支持层次状态（巡逻包含子状态）

### 5.4 用户界面

UI组件状态：
- 普通 -> 悬停 -> 按下 -> 禁用
- 支持动画和过渡效果
