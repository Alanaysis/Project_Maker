# 需求分析文档

## 1. 项目概述

### 1.1 项目目标
构建一个轻量级CDN服务，用于学习CDN核心原理和架构设计。

### 1.2 核心价值
- 理解CDN工作原理
- 掌握缓存策略实现
- 学习负载均衡算法
- 实践高性能网络编程

## 2. 用户画像

### 2.1 主要用户
**学习者**
- 背景：有编程基础，想深入理解CDN技术
- 需求：清晰的代码结构，详细的注释，循序渐进的学习路径
- 痛点：现有CDN项目过于复杂，难以入门

### 2.2 次要用户
**开发者**
- 背景：需要在项目中集成简单CDN功能
- 需求：易于集成，配置简单，性能可靠
- 痛点：商用CDN成本高，配置复杂

## 3. 功能需求

### 3.1 核心功能

#### F1: HTTP代理服务
**优先级**: P0
**描述**: 接收HTTP请求，转发到源站或返回缓存内容

**功能点**：
- 支持GET请求代理
- 支持HTTP/1.1
- 支持请求头透传
- 支持响应头透传

**验收标准**：
- 能够正确代理HTTP请求
- 响应状态码正确
- 响应内容完整

#### F2: 内容缓存
**优先级**: P0
**描述**: 缓存源站响应内容，减少回源请求

**功能点**：
- 基于LRU的缓存淘汰
- 可配置缓存大小
- 可配置缓存TTL
- 缓存命中/未命中统计

**验收标准**：
- 缓存命中时直接返回，不回源
- 缓存过期后重新回源
- LRU淘汰逻辑正确

#### F3: 回源机制
**优先级**: P0
**描述**: 缓存未命中时从源站获取内容

**功能点**：
- 从源站获取内容
- 支持超时控制
- 支持重试机制
- 回源后更新缓存

**验收标准**：
- 缓存未命中时正确回源
- 回源失败时返回错误
- 回源成功后更新缓存

#### F4: 智能调度
**优先级**: P1
**描述**: 根据策略选择最优节点

**功能点**：
- 轮询调度
- 加权轮询调度
- 最少连接调度
- 节点健康检查

**验收标准**：
- 请求均匀分发到各节点
- 权重高的节点处理更多请求
- 故障节点自动剔除

### 3.2 辅助功能

#### F5: 健康检查
**优先级**: P1
**描述**: 定期检查节点健康状态

**功能点**：
- HTTP健康检查
- TCP健康检查
- 可配置检查间隔
- 故障节点自动恢复

#### F6: 日志记录
**优先级**: P2
**记录请求和响应信息

**功能点**：
- 访问日志
- 错误日志
- 缓存命中日志
- 可配置日志级别

#### F7: 监控指标
**优先级**: P2
**描述**: 暴露运行时指标

**功能点**：
- 请求计数
- 缓存命中率
- 响应时间
- 错误率

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 并发连接数 | > 1000 | 同时处理的请求数 |
| 响应时间 | < 50ms | 缓存命中时的响应时间 |
| 缓存命中率 | > 90% | 缓存有效时的命中率 |
| 回源延迟 | < 200ms | 从源站获取内容的时间 |

### 4.2 可靠性需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 可用性 | > 99.9% | 系统正常运行时间 |
| 故障恢复 | < 30s | 节点故障后的恢复时间 |
| 数据一致性 | 最终一致 | 缓存与源站的一致性 |

### 4.3 可扩展性需求

- 支持水平扩展
- 支持动态添加节点
- 支持配置热更新

### 4.4 安全需求

- 支持HTTPS（可选）
- 支持访问控制
- 防止缓存投毒

## 5. 技术约束

### 5.1 开发环境
- 操作系统：Linux/macOS
- Go版本：1.21+
- 依赖管理：Go Modules

### 5.2 运行环境
- 内存：最少1GB
- CPU：最少2核
- 磁盘：根据缓存大小

### 5.3 协议支持
- HTTP/1.1
- HTTPS（可选）
- WebSocket（可选）

## 6. 接口设计

### 6.1 HTTP代理接口

```
GET /{path} HTTP/1.1
Host: cdn.example.com
```

**响应**：
```
HTTP/1.1 200 OK
Content-Type: text/html
X-Cache: HIT  # 或 MISS
X-Cache-TTL: 3600

<html>...</html>
```

### 6.2 管理接口

#### 获取缓存统计
```
GET /admin/cache/stats
```

**响应**：
```json
{
  "hits": 1234,
  "misses": 567,
  "hit_rate": 0.685,
  "size": 1024,
  "max_size": 10240
}
```

#### 清除缓存
```
POST /admin/cache/purge
Content-Type: application/json

{
  "path": "/test.html"
}
```

#### 获取节点状态
```
GET /admin/nodes/status
```

**响应**：
```json
{
  "nodes": [
    {
      "id": "node-1",
      "address": "192.168.1.1:8080",
      "status": "healthy",
      "connections": 10,
      "weight": 1
    }
  ]
}
```

## 7. 数据模型

### 7.1 缓存条目

```go
type CacheEntry struct {
    Key        string
    Value      []byte
    Headers    http.Header
    StatusCode int
    CreatedAt  time.Time
    ExpiresAt  time.Time
    AccessCount int64
}
```

### 7.2 节点信息

```go
type Node struct {
    ID        string
    Address   string
    Weight    int
    Status    string  // healthy, unhealthy, unknown
    Connections int
    LastCheck time.Time
}
```

### 7.3 请求上下文

```go
type RequestContext struct {
    Request   *http.Request
    ClientIP  string
    Path      string
    CacheKey  string
    StartTime time.Time
}
```

## 8. 用例场景

### 8.1 场景1：首次访问（缓存未命中）

```
用户 → CDN → 缓存未命中 → 回源 → 获取内容 → 更新缓存 → 返回用户
```

### 8.2 场景2：重复访问（缓存命中）

```
用户 → CDN → 缓存命中 → 直接返回
```

### 8.3 场景3：缓存过期

```
用户 → CDN → 缓存过期 → 回源 → 获取新内容 → 更新缓存 → 返回用户
```

### 8.4 场景4：源站故障

```
用户 → CDN → 缓存未命中 → 回源失败 → 返回错误（或返回过期缓存）
```

## 9. 验收标准

### 9.1 功能验收

- [ ] 能够代理HTTP请求
- [ ] 能够缓存响应内容
- [ ] 能够回源获取内容
- [ ] 能够实现LRU淘汰
- [ ] 能够进行健康检查

### 9.2 性能验收

- [ ] 并发连接数 > 1000
- [ ] 缓存命中响应时间 < 50ms
- [ ] 缓存命中率 > 90%

### 9.3 可靠性验收

- [ ] 故障节点自动剔除
- [ ] 故障恢复后自动加入
- [ ] 无内存泄漏

## 10. 优先级规划

### Phase 1: MVP（最小可行产品）
- HTTP代理服务
- LRU缓存
- 基本回源机制

### Phase 2: 增强功能
- 智能调度
- 健康检查
- 监控指标

### Phase 3: 高级特性
- HTTPS支持
- 缓存预热
- 动态配置
- 分布式缓存

## 11. 风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 性能不达标 | 高 | 中 | 性能测试和优化 |
| 缓存一致性问题 | 中 | 高 | 合理的TTL策略 |
| 源站故障 | 高 | 中 | 缓存过期内容服务 |
| 内存溢出 | 高 | 低 | 缓存大小限制 |

## 12. 总结

本项目旨在构建一个学习型CDN服务，通过实现核心功能来理解CDN的工作原理。项目采用Go语言开发，注重代码清晰度和可学习性，同时保证基本的性能和可靠性。
