# 市场调研

## C++ 三方库生态概览

### 生态现状

C++ 三方库生态成熟且多样化，主要特点：

1. **历史悠久** - 许多库已有 20+ 年历史（如 Boost、libcurl）
2. **企业支持** - Google、Facebook、Microsoft 等大厂主导多个重要项目
3. **社区活跃** - GitHub 上有大量高质量开源项目
4. **标准影响** - 许多三方库特性最终进入 C++ 标准

### 主要来源

| 来源 | 代表项目 | 特点 |
|------|----------|------|
| Google | Protobuf, Abseil, GTest, FlatBuffers | 工程化、文档完善 |
| Facebook | Folly, HHVM | 高性能、前沿技术 |
| Boost | Boost.Asio, Boost.Container | 标准化试验田 |
| 个人项目 | spdlog, nlohmann/json, Catch2 | 创新、易用 |
| 学术项目 | Eigen, Armadillo | 算法先进 |

## 库选择标准

### 1. 成熟度评估

```
评估维度：
├── 版本历史
│   ├── 稳定版本数量
│   ├── 发布频率
│   └── 向后兼容性
├── 社区活跃度
│   ├── GitHub Stars/Forks
│   ├── Issue 响应时间
│   └── PR 合并速度
├── 文档质量
│   ├── API 文档完整性
│   ├── 教程和示例
│   └── 迁移指南
└── 测试覆盖
    ├── 单元测试覆盖率
    ├── CI/CD 配置
    └── 平台支持
```

### 2. 性能考量

关键指标：
- **编译时间** - 头文件库 vs 预编译库
- **运行时性能** - 内存分配、缓存友好性
- **二进制大小** - 静态链接 vs 动态链接
- **依赖复杂度** - 传递依赖数量

### 3. 易用性评估

```
易用性维度：
├── API 设计
│   ├── 一致性
│   ├── 命名规范
│   └── 错误处理
├── 集成难度
│   ├── 构建系统支持
│   ├── 包管理器支持
│   └── 平台兼容性
└── 学习曲线
    ├── 入门门槛
    ├── 进阶难度
    └── 社区资源
```

## 版本管理策略

### 语义化版本

```
主版本.次版本.修订号 (Major.Minor.Patch)

示例：
- 1.0.0 - 首个稳定版本
- 1.1.0 - 新增功能，向后兼容
- 1.1.1 - Bug 修复
- 2.0.0 - 不兼容的 API 变更
```

### 版本选择建议

| 场景 | 建议 | 原因 |
|------|------|------|
| 生产环境 | 最新稳定版 | 安全性和稳定性 |
| 新项目 | 最新稳定版 | 获取最新特性 |
| 遗留项目 | 次新版本 | 平衡稳定性和特性 |
| 实验性项目 | 最新开发版 | 尝试新特性 |

### 依赖管理最佳实践

```cmake
# CMake FetchContent 示例
include(FetchContent)

FetchContent_Declare(
  fmt
  GIT_REPOSITORY https://github.com/fmtlib/fmt.git
  GIT_TAG 10.2.1  # 锁定版本
)

FetchContent_MakeAvailable(fmt)
```

## 各领域库对比

### 容器库对比

| 特性 | Boost.Container | Abseil | Folly | EASTL |
|------|-----------------|--------|-------|-------|
| 内存效率 | 高 | 高 | 高 | 极高 |
| 编译速度 | 慢 | 中 | 慢 | 快 |
| API 一致性 | Boost 风格 | 现代 C++ | 现代 C++ | STL 风格 |
| 适用场景 | 通用 | 通用 | 高性能 | 游戏 |

### 序列化库对比

| 特性 | Protobuf | FlatBuffers | Cap'n Proto | JSON |
|------|----------|-------------|-------------|------|
| 性能 | 高 | 极高 | 极高 | 中 |
| 代码生成 | 需要 | 需要 | 需要 | 不需要 |
| 跨语言 | 是 | 是 | 是 | 是 |
| Schema | .proto | .fbs | .capnp | 无 |
| 适用场景 | RPC/存储 | 游戏/实时 | 低延迟 | Web/API |

### 网络库对比

| 特性 | Boost.Asio | cpp-httplib | CPR | libcurl |
|------|------------|-------------|-----|---------|
| 复杂度 | 高 | 低 | 低 | 中 |
| 功能 | 完整 | HTTP | HTTP | 完整 |
| 异步支持 | 是 | 否 | 否 | 是 |
| 适用场景 | 底层网络 | 简单 HTTP | 简单 HTTP | 通用 |

### 测试框架对比

| 特性 | GTest | Catch2 | doctest |
|------|-------|--------|---------|
| 编译速度 | 中 | 慢 | 极快 |
| 功能 | 完整 | 完整 | 基础 |
| 学习曲线 | 平缓 | 平缓 | 极平缓 |
| 宏使用 | 多 | 少 | 少 |
| 适用场景 | 企业项目 | 现代项目 | 小型项目 |

## 推荐组合

### Web 后端开发

```
序列化: Protobuf + JSON
网络: Boost.Asio 或 cpp-httplib
日志: spdlog
测试: GTest + GMock
配置: nlohmann/json
```

### 游戏开发

```
容器: EASTL
序列化: FlatBuffers
数学: Eigen
图形: SDL/SFML
并发: Intel TBB
```

### 科学计算

```
数学: Eigen + Armadillo
并发: Intel TBB + Taskflow
日志: spdlog
测试: GTest
构建: CMake + vcpkg
```

### 嵌入式/性能敏感

```
容器: EASTL
序列化: Cap'n Proto
日志: spdlog (异步模式)
内存: 自定义分配器
构建: CMake + Conan
```

## 总结

选择三方库时应考虑：

1. **项目需求** - 功能是否满足
2. **团队熟悉度** - 学习成本
3. **长期维护** - 社区活跃度
4. **性能要求** - 运行时和编译时
5. **许可证** - 是否兼容项目许可证

建议从成熟、文档完善的库开始，逐步引入更专业的库。
