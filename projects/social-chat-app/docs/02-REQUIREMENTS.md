# 需求分析

## 1. 项目概述

### 1.1 项目目标

构建一个支持实时消息、群聊、文件传输的社交聊天系统，作为学习即时通讯架构的实践项目。

### 1.2 核心价值

- **学习价值**：深入理解 IM 系统的核心原理
- **实用价值**：可作为小型团队的内部通讯工具
- **技术价值**：掌握 WebSocket、消息队列、加密等技术

## 2. 用户画像

### 2.1 主要用户：开发者/学习者

**特征**：
- 对即时通讯技术感兴趣
- 希望理解 IM 系统的内部工作原理
- 有一定的编程基础（Go 或其他语言）

**使用场景**：
- 学习 WebSocket 编程
- 理解消息队列的应用
- 实践端到端加密

**需求**：
- 清晰的代码结构
- 详细的文档说明
- 可运行的示例程序

### 2.2 次要用户：小型团队

**特征**：
- 3-20 人的小团队
- 需要简单的内部通讯工具
- 不想依赖第三方服务

**使用场景**：
- 日常工作沟通
- 文件分享
- 项目讨论

**需求**：
- 稳定可靠
- 易于部署
- 基本的安全性

## 3. 功能需求

### 3.1 用户管理（P0 - 必须实现）

#### 3.1.1 用户注册
- **描述**：新用户创建账号
- **输入**：用户名、密码、邮箱（可选）
- **输出**：注册成功/失败
- **规则**：
  - 用户名唯一，3-20 字符
  - 密码长度 >= 8 字符
  - 邮箱格式验证（如果提供）

#### 3.1.2 用户登录
- **描述**：用户登录系统
- **输入**：用户名、密码
- **输出**：JWT Token
- **规则**：
  - 验证用户名和密码
  - 返回有效的 JWT Token
  - Token 有效期 24 小时

#### 3.1.3 用户信息
- **描述**：获取/更新用户信息
- **操作**：
  - GET /api/user/:id - 获取用户信息
  - PUT /api/user/:id - 更新用户信息
- **字段**：昵称、头像、签名

### 3.2 单聊（P0 - 必须实现）

#### 3.2.1 发送消息
- **描述**：用户向另一个用户发送消息
- **输入**：接收者 ID、消息内容
- **输出**：消息发送状态
- **规则**：
  - 消息类型：文本、图片、文件
  - 消息状态：已发送、已送达、已读
  - 支持消息撤回（2 分钟内）

#### 3.2.2 接收消息
- **描述**：用户接收其他用户的消息
- **触发**：WebSocket 推送
- **处理**：
  - 在线用户：实时推送
  - 离线用户：存储后上线拉取

#### 3.2.3 消息历史
- **描述**：查看与某用户的历史消息
- **输入**：对方用户 ID、时间范围、分页参数
- **输出**：消息列表
- **规则**：
  - 支持按时间分页
  - 最近 100 条消息免费查询
  - 更早的消息需要付费（扩展功能）

### 3.3 群聊（P1 - 应该实现）

#### 3.3.1 创建群组
- **描述**：创建一个新的群组
- **输入**：群名称、群成员
- **输出**：群组 ID
- **规则**：
  - 群名称：2-50 字符
  - 群成员上限：500 人
  - 创建者自动成为群主

#### 3.3.2 群组消息
- **描述**：在群组中发送消息
- **输入**：群组 ID、消息内容
- **输出**：消息发送状态
- **规则**：
  - 只有群成员可以发送消息
  - 消息推送给所有在线群成员
  - 离线成员上线后拉取

#### 3.3.3 群组管理
- **操作**：
  - 邀请成员
  - 移除成员
  - 退出群组
  - 修改群信息
  - 转让群主

### 3.4 文件传输（P1 - 应该实现）

#### 3.4.1 上传文件
- **描述**：上传文件到服务器
- **输入**：文件内容、文件名
- **输出**：文件 URL
- **规则**：
  - 文件大小限制：50MB
  - 支持格式：图片、文档、压缩包
  - 文件存储在本地或 OSS

#### 3.4.2 下载文件
- **描述**：下载聊天中的文件
- **输入**：文件 URL
- **输出**：文件内容
- **规则**：
  - 只有聊天参与者可以下载
  - 支持断点续传（扩展功能）

### 3.5 离线消息（P0 - 必须实现）

#### 3.5.1 消息存储
- **描述**：用户离线时存储消息
- **触发**：接收方不在线
- **处理**：
  - 消息存储到数据库
  - 标记为未读
  - 更新未读计数

#### 3.5.2 消息同步
- **描述**：用户上线后同步离线消息
- **触发**：用户登录成功
- **处理**：
  - 查询所有未读消息
  - 按时间顺序推送
  - 更新消息状态为已送达

### 3.6 用户状态（P1 - 应该实现）

#### 3.6.1 在线状态
- **描述**：实时更新用户在线状态
- **状态**：在线、离线、忙碌、离开
- **实现**：
  - WebSocket 连接时标记在线
  - 断开连接时标记离线
  - 心跳检测（30 秒无响应标记离线）

#### 3.6.2 输入状态
- **描述**：显示对方正在输入
- **触发**：用户开始输入
- **处理**：
  - 发送 typing 事件
  - 接收方显示 "正在输入..."
  - 3 秒后自动消失

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 消息延迟 | < 100ms | 本地网络环境 |
| 并发连接 | 1000+ | 单服务器实例 |
| 消息吞吐 | 10000 msg/s | 峰值处理能力 |
| 数据库查询 | < 50ms | P95 延迟 |

### 4.2 可靠性需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 消息送达率 | 99.9% | 不丢失消息 |
| 系统可用性 | 99.9% | 年停机时间 < 8.76 小时 |
| 数据持久化 | 99.999% | 数据不丢失 |

### 4.3 安全需求

- **传输加密**：使用 WSS（WebSocket Secure）
- **身份认证**：JWT Token 认证
- **数据加密**：端到端加密（E2EE）
- **访问控制**：基于角色的权限控制
- **输入验证**：防止 XSS、SQL 注入

### 4.4 可扩展性需求

- 支持水平扩展（多服务器实例）
- 支持数据库分片
- 支持消息队列集群

## 5. 用户故事

### 5.1 用户注册与登录

```
作为一名新用户
我想要注册一个账号
以便使用聊天功能

验收标准：
- 可以使用用户名和密码注册
- 注册成功后自动登录
- 收到注册成功的确认消息
```

```
作为一名已注册用户
我想要登录我的账号
以便查看我的消息

验收标准：
- 可以使用用户名和密码登录
- 登录后看到我的联系人列表
- 收到离线期间的消息
```

### 5.2 发送和接收消息

```
作为一名已登录用户
我想要给朋友发送消息
以便与他交流

验收标准：
- 可以选择一个联系人
- 可以输入文本消息
- 消息发送后显示在聊天窗口
- 对方在线时实时收到消息
```

```
作为一名已登录用户
我想要接收朋友的消息
以便及时回复

验收标准：
- 在线时实时收到新消息
- 收到新消息时有提示
- 可以查看消息内容
```

### 5.3 离线消息

```
作为一名用户
我想要在上线后看到离线期间的消息
以便了解错过了什么

验收标准：
- 登录后收到所有离线消息
- 离线消息按时间顺序显示
- 未读消息有明显标记
```

### 5.4 群聊

```
作为一名用户
我想要创建一个群组
以便与多人交流

验收标准：
- 可以创建新群组
- 可以邀请好友加入
- 群组内可以发送消息
- 所有群成员都能收到消息
```

## 6. 约束条件

### 6.1 技术约束

- 使用 Go 语言实现
- 使用 WebSocket 协议
- 支持 SQLite 和 PostgreSQL
- 支持 Redis（可选）

### 6.2 时间约束

- 第一阶段（2 周）：用户管理和单聊
- 第二阶段（2 周）：离线消息和状态同步
- 第三阶段（2 周）：群聊和文件传输
- 第四阶段（2 周）：端到端加密和优化

### 6.3 资源约束

- 单服务器部署
- 本地文件存储
- 有限的测试资源
