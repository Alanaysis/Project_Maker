# 开发记录

## 1. 开发环境

- 操作系统: Linux (Ubuntu)
- 编译器: GCC (C11 标准)
- 构建工具: Make
- 调试器: GDB

## 2. 开发过程

### 2.1 阶段一: 基础框架
- 定义公共类型和常量 (riscv.h)
- 实现内存模型 (memory.h/c)
- 实现指令解码器 (decoder.h/c)

### 2.2 阶段二: 执行引擎
- 实现指令执行器 (executor.h/c)
  - R-type ALU 指令
  - I-type ALU 指令
  - Load/Store 指令
  - Branch 指令
  - Jump 指令
  - U-type 指令
  - System 指令
- 实现 CPU 模块 (cpu.h/c)
  - 取指-解码-执行循环
  - 寄存器管理

### 2.3 阶段三: 汇编器
- 实现简单汇编器 (assembler.h/c)
  - 寄存器名称解析
  - 立即数解析
  - 各类型指令编码
  - 标签支持 (两遍汇编)

### 2.4 阶段四: 测试和文档
- 编写单元测试
- 编写集成测试
- 创建示例程序
- 编写文档

## 3. 关键技术决策

### 3.1 语言选择
选择 C 语言而不是 Rust，原因:
- 更贴近底层，更容易理解指令执行
- 编译简单，无外部依赖
- 适合教学用途

### 3.2 内存模型
- 使用平坦内存模型 (flat memory)
- 基地址可配置 (默认 0x80000000)
- 1MB 默认大小

### 3.3 指令解码
- 一次性解码所有字段
- 立即数在解码时就完成符号扩展
- 使用枚举类型表示操作码和功能码

### 3.4 汇编器设计
- 两遍汇编: 第一遍收集标签，第二遍生成代码
- 标签使用相对偏移 (相对于当前 PC)
- 支持 ABI 和数字两种寄存器名称

## 4. 遇到的问题和解决方案

### 4.1 SUB 指令编码错误
**问题**: SUB 指令执行结果为 0
**原因**: ALU 功能码计算错误，`funct7 << 3` 应为 `funct7 << 3`，但 ALU_SUB 定义为 `0x400` 而非 `0x100`
**解决**: 修正 ALU_SUB 为 `0x100` (0x20 << 3)

### 4.2 标签解析失败
**问题**: 带标签的分支指令被跳过
**原因**: 符号表地址使用 `i32` 类型，0x80000000 溢出为负数
**解决**: 改用 `u32` 存储地址，`bool` 返回查找结果

### 4.3 LUI 立即数编码
**问题**: LUI 编码结果不正确
**原因**: 用户提供的立即数需要左移 12 位
**解决**: 在汇编器中对 LUI/AUIPC 立即数进行 `imm << 12`

### 4.4 Load/Store 测试地址越界
**问题**: Load/Store 测试返回 0
**原因**: 测试使用的数据地址 0x10000100 超出 1MB 内存范围
**解决**: 改用 0x800FF000 (在内存范围内)

### 4.5 BLT 循环边界问题
**问题**: 求和 1+2+...+10 结果为 45
**原因**: BLT 是 "小于则跳转"，用 10 作为上界时循环到 9 就停止
**解决**: 使用 11 作为上界
