# 防火墙调研报告

## 1. 防火墙概述

### 1.1 什么是防火墙

防火墙是一种网络安全系统，用于监控和控制进出网络的流量。它基于预定义的安全规则，决定允许或阻止特定的数据包通过。

### 1.2 防火墙的类型

| 类型 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 包过滤防火墙 | 检查数据包头部信息 | 速度快、简单 | 无法检测应用层攻击 |
| 状态检测防火墙 | 跟踪连接状态 | 安全性高 | 资源消耗较大 |
| 应用层防火墙 | 检查应用层数据 | 安全性最高 | 速度慢、复杂 |
| 下一代防火墙 | 综合多种功能 | 功能全面 | 成本高 |

### 1.3 防火墙的工作层次

```
应用层 (Layer 7)     ← 应用层防火墙
传输层 (Layer 4)     ← 状态检测防火墙
网络层 (Layer 3)     ← 包过滤防火墙
数据链路层 (Layer 2)
物理层 (Layer 1)
```

## 2. 包过滤技术

### 2.1 包过滤原理

包过滤防火墙检查每个数据包的以下信息：

- **源 IP 地址**：数据包来自哪里
- **目的 IP 地址**：数据包要去哪里
- **源端口**：发送方的端口号
- **目的端口**：接收方的端口号
- **协议类型**：TCP、UDP、ICMP 等

### 2.2 过滤规则

规则通常采用"默认拒绝"策略：

```
规则 1: 允许来自 192.168.1.0/24 的 HTTP 流量 (端口 80)
规则 2: 允许来自 192.168.1.0/24 的 HTTPS 流量 (端口 443)
规则 3: 阻止所有其他流量
```

### 2.3 CIDR 表示法

CIDR (Classless Inter-Domain Routing) 用于表示 IP 地址范围：

- `192.168.1.0/24` 表示 192.168.1.0 到 192.168.1.255
- `10.0.0.0/8` 表示 10.0.0.0 到 10.255.255.255
- `172.16.0.0/12` 表示 172.16.0.0 到 172.31.255.255

## 3. 状态检测技术

### 3.1 状态检测原理

状态检测防火墙不仅检查单个数据包，还跟踪连接的状态：

1. **NEW**：新连接（SYN 包）
2. **ESTABLISHED**：已建立的连接
3. **RELATED**：相关连接（如 FTP 数据连接）
4. **INVALID**：无效数据包

### 3.2 TCP 状态机

```
CLOSED → SYN_SENT → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
    ↓
SYN_RECEIVED → ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
```

### 3.3 连接跟踪表

连接跟踪表存储活动连接的信息：

| 字段 | 说明 |
|------|------|
| 源 IP | 发送方 IP 地址 |
| 源端口 | 发送方端口 |
| 目的 IP | 接收方 IP 地址 |
| 目的端口 | 接收方端口 |
| 协议 | TCP/UDP/ICMP |
| 状态 | 连接状态 |
| 超时 | 连接超时时间 |

## 4. 入侵检测技术

### 4.1 入侵检测系统 (IDS)

IDS 用于检测网络中的恶意活动：

- **基于签名的检测**：匹配已知攻击模式
- **基于异常的检测**：检测偏离正常行为的活动
- **基于状态的检测**：跟踪连接状态变化

### 4.2 常见攻击检测

#### SYN Flood 攻击

攻击者发送大量 SYN 包，耗尽服务器资源：

```
检测方法：统计来自同一 IP 的 SYN 包数量
阈值：每分钟超过 1000 个 SYN 包
```

#### 端口扫描

攻击者扫描目标主机的开放端口：

```
检测方法：统计来自同一 IP 的不同端口访问数
阈值：每分钟访问超过 20 个不同端口
```

#### 连接频率异常

异常的高频连接尝试：

```
检测方法：统计来自同一 IP 的连接尝试次数
阈值：每分钟超过 100 次连接尝试
```

## 5. 防火墙架构设计

### 5.1 模块化设计

```
┌─────────────────────────────────────────────┐
│                 防火墙引擎                    │
├─────────────────────────────────────────────┤
│  包解析模块  │  规则引擎  │  状态检测模块     │
├─────────────────────────────────────────────┤
│  日志模块    │  入侵检测  │  配置管理模块     │
└─────────────────────────────────────────────┘
```

### 5.2 数据流

```
数据包输入
    ↓
包解析
    ↓
状态检测 → 已建立连接？ → 是 → 允许
    ↓ 否
规则匹配 → 匹配规则？ → 是 → 执行动作
    ↓ 否
默认动作
    ↓
日志记录
```

### 5.3 规则匹配算法

#### 线性扫描

最简单的算法，按顺序检查每条规则：

```python
for rule in rules:
    if rule.matches(packet):
        return rule.action
return default_action
```

时间复杂度：O(n)，其中 n 是规则数量

#### 优化方案

1. **按协议分组**：先检查协议，再匹配规则
2. **使用哈希表**：对精确匹配的规则使用哈希表
3. **规则排序**：将高频匹配的规则放在前面

## 6. Python 实现考虑

### 6.1 为什么选择 Python

| 优点 | 缺点 |
|------|------|
| 易于学习和理解 | 性能较低 |
| 丰富的标准库 | 不适合生产环境 |
| 良好的可读性 | GIL 限制并发 |
| 快速原型开发 | 内存消耗较大 |

### 6.2 性能优化

1. **使用 dataclass**：减少样板代码
2. **使用枚举**：提高代码可读性
3. **使用类型提示**：提高代码质量
4. **使用锁**：保证线程安全

### 6.3 依赖库

- **PyYAML**：YAML 配置文件解析
- **pytest**：测试框架
- **scapy**：高级数据包处理（可选）
