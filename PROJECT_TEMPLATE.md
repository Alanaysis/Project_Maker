# 📋 项目模板规范 / Project Template Specification

[English](#english) | [中文](#中文)

---

## 中文

### 概述

本文档定义了学习型项目的标准结构和规范。每个项目必须遵循此模板，确保一致性和学习价值。

### 核心原则

1. **最小可执行**：只包含最核心的功能循环
2. **学习导向**：重点难点突出，值得思考指明
3. **工程规范**：完整的文档和测试
4. **产品思维**：包含市场调研和用户分析

---

### 项目目录结构

```
project-xxx/
├── README.md                    # 项目总览（必须）
├── docs/                        # 文档目录（必须）
│   ├── 01-RESEARCH.md          # 市场调研（必须）
│   ├── 02-REQUIREMENTS.md      # 需求分析（必须）
│   ├── 03-DESIGN.md            # 技术设计（必须）
│   ├── 04-PRODUCT.md           # 产品思维（必须）
│   └── 05-DEVELOPMENT.md       # 开发手册（必须）
├── src/                         # 源代码（必须）
│   └── ...
├── tests/                       # 测试（必须）
│   └── ...
├── examples/                    # 使用示例（必须）
│   └── ...
├── LEARNING_NOTES.md            # 学习笔记模板（必须）
├── CHANGELOG.md                 # 变更记录（建议）
└── requirements.txt             # 依赖清单（如需要）
```

## 1. 目标用户

### 用户画像

**主要用户**：
- 身份：初学者/中级开发者
- 痛点：想学习 XXX，但找不到好的入门项目

**次要用户**：
- 身份：面试者
- 痛点：需要理解 XXX 的核心原理
- 需求：能快速理解并复述的项目

## 2. 使用场景

### 场景1：学习入门
```
用户 → 阅读 README → 运行示例 → 理解核心 → 深入学习
```

### 场景2：面试准备
```
用户 → 阅读设计文档 → 理解决策 → 能够复述
```

### 场景3：项目参考
```
用户 → 查看实现 → 提取模式 → 应用到自己的项目
```

## 3. 用户吸引力

### 为什么用户会选择这个项目？

1. **清晰的学习路径**
   - 不是一堆代码，而是有组织的学习材料
   - 从简到难，循序渐进

2. **深度而非广度**
   - 不追求功能多，追求理解深
   - 每个决策都有解释

3. **实用性**
   - 代码可以运行
   - 示例可以直接用

4. **扩展性**
   - 看懂核心后，可以自己扩展
   - 提供扩展指南

## 4. 竞品对比

| 维度 | 本项目 | 竞品A | 竞品B |
|------|--------|-------|-------|
| 学习曲线 | 平缓 | 陡峭 | 中等 |

| 代码可运行 | ✅ | ❌ | ✅ |
| 设计解释 | ✅ | ❌ | ❌ |
| 扩展性 | 高 | 低 | 中 |
| 市场调研 | ✅ | ❌ | ❌ |
| 产品思维 | ✅ | ❌ | ❌ |

## 5. 差异化策略

**核心差异**：不只是代码，而是完整的**学习体验**

1. **调研先行**
   - 告诉你别人怎么做，为什么
   - 帮你建立全局视野

2. **设计透明**
   - 告诉你为什么这么设计
   - 理解决策背后的权衡

3. **产品视角**
   - 告诉你如何吸引用户
   - 培养产品思维

4. **学习导向**
   - 重点难点突出，值得思考指明
   - 提供学习笔记模板

1. **内容质量**
   - 高质量的文档和代码
   - 真正有深度的分析
